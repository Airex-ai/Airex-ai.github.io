自动驾驶硬件概述

感知传感器

  • 摄像头：广泛用于物体识别和物体追踪场景，如车道线检测、交通灯识别
  • 激光雷达：用于障碍物识别、绘制地图、辅助定位
  • 毫米波雷达：阴雨天、雾霾天能够辅助感知获取物体的位置和速度
  • 超声波：近处高敏感度传感器，常用于作为安全冗余设备检测车辆的碰撞安全问题

定位系统传感器

  • IMU：实时测量自身的姿态，200hz或更高。包含了三个单轴的加速度计和三个单轴的陀螺仪，加速度计检测物体在载体坐标系统独立三轴的加速度信号，而陀螺检测载体相对于导航坐标系的角速度的信号

  • GNSS：GPS，无人车一般使用RTK（载波相位差分技术）进行定位，频率相对较低，10hz左右

车载计算单元（IPC）

  • 高效连接计算单元内部各计算设备，连接外部传感器的信息输入和存储
  • 冗余设计，以防止单点故障
  • 需要考虑整体的车规、电磁干扰和振动方面的设计以及ISO-26262标准的要求
  • ISO-26262：一个硬件达到了ASIL D级别的要求，那么它的故障率是10FIT，即10亿个小时里面出一次故障，汽车行业在安全方面可以做到的极限（迭代速度慢）

车辆线控系统

  • 自动驾驶线控系统：汽车的控制是一些简单命令完成的，而不是由物理操作完成的，这一部分相当于人的手和脚
  • 传统汽车的这些控制由液压系统和真空助力泵协助完成，自动驾驶汽车的线控需要用电控化的零部件来完成，如电子液压制动系统

操作系统

  • RTOS：实时操作系统
    • QNX：类Unix系统，具有强实时性，符合车规级的实时操作系统
    • RT Linux：通过添加Linux内核补丁，通过软实时进行监控
  • Framework：
    • ROS（机器人操作系统）
    • YARP、Microsoft Robotics

高清地图 HD Map（High Dimensional）

不同于导航地图，最大的特点就是高维度和高精度。具备道路网的精确三维表征，如交叉路口布局和路标位置。具备地图语义信息，如道路的速度限制，左转车道开始的位置。导航地图智能达到米级精度，高精地图需要能够达到厘米级的精度

  • 高精地图提供其他level 4模块的数据支持
  • 提供了很多准确的静态物体的信息
  • 定位用于计算相对位置
  • 帮助传感器缩小检测范围，缩小感兴趣区域ROI
  • 计算道路导航信息
  • 帮助车辆识别车道的确切中心线

定位

INS（惯性导航系统）：IMU获取自身的状态（加速度和角速度）后通过状态矩阵递推下一时刻位置，但是如果没有校正信息的话，这种状态递推会随着时间不断累计误差，导致最终位置发散

RTK（载波相位差分系统）：RTK多加了一个静止基站，它同样也收到定位卫星的信号。无人车与RTK像个不太远的情况下，对二者之间的干扰信号用差分抹平

RTK通过较低的更新频率提供相对准确的位置信息，INS则以较高的频率提供准确性交叉的姿态信息。通过使用Kalman Filter整合两类数据获取其各自有时，合并提供出高准确性的实时信息

几何定位

通过使用激光雷达、摄像头、高精地图进行定位。利用激光雷达或者图像信息，可以通过物体匹配来对汽车进行定位。将检测的数据与预先存在的高精地图之间匹配，通过这种比较可获取汽车在高精地图上的全球位置和行驶方向

迭代最近点（ICP）\ 直方图滤波（Histogram Filter）

感知

基本任务是：检测，分类，跟踪，分割。即找出物体在环境中的位置，明确对象是什么，随时间的持续观察移动物体并保持一致，图像中的每个像素与语义类别进行匹配，如道路、汽车、天空、边界清晰

预测

预测具备实时性和准确性，又分为

  • 基于状态进行预测：Kalman Filter、Particle Filter
  • 基于车道序列进行预测：通过机器学习模型化简为分类问题
  • 行人预测：无人车需要非常重视安全问题。其中人的安全最为重要，而行人的意图变化确实最难预测的，也是约束最少的

决策规划

  • 导航线路规划和精细轨迹表述
  • 最优路径搜索：由于其他软件模块已经将不确定性进行了最大程度的消除，而最终决策规划模块又是对稳定性要求极高的模块，因此可以通过数学上的最优路径求解出确定解，遍历最优解是非常耗时的
  • 需要考虑车辆的体感和安全性

控制

以目标路径、车辆状态为输入信息，输出方向盘、油门的信息，实现对无人车的控制。

控制是对整个驾驶最后的保障，因此需要在任何情况下对准确性、稳定性和时效性要求都非常高。

传统的控制算法PID可以满足车辆控制要求，但是考虑到体感和一些极限情况，控制算法优化也是目前无人车的一个持续探讨的问题，如LQR、MPC等

ROS

ROS：The Robot Operating System

特点：点对点设计、分布式设计、多语言（python、C++）、轻量级、免费且开源、社区完善

ROS = Plumbing + Tools + Capabilities + Ecosystem

ROS不是传统意义上的一种操作系统，而是一种系统软件框架，该框架使用了流行的面向服务（SOA）的软件技术，通过网络协议（类似于HTTP协议）将节点间数据通信解耦。这样就能够轻松地集成不同语言不同功能的代码。

ROS不仅是一个函数库，除包含客户端外，还包括一个中心服务器、一系列命令行工具、图形化界面工具以及编译环境。

通信

  • 消息以一种publish/subscribe的方式传递

  • 节点可以在给定的主题中发布/订阅信息

  • 一个节点可以订阅/发布不同的主题

  • 允许多个节点订阅/发布同一个主题

  • 订阅节点和发布节点并不知道相互之间的存在

Packages & Catkin Workspaces

packages：Package是ROS系统中最底层最基本的 组织，存放各种文件：库、工具、可执行文件等

Catkin Workspaces：包的顶层工作目录，一个catkin wordspace包含一个工程下多个ros package

package.xml：每个包的描述文件，都需要放置在包的根目录下，将包的名字/版本/作者/维护者/依赖关系进行说明

CmakeList.txt：定义了一个包的编译指令

  • cmake不会找package.xml文件，根据cmakelists.txt文件编译需要清晰指出头文件和库文件的指向

Catkin

ROS的编译系统，对CMake进行扩展，适合ROS进行大型项目，多包，多节点的场景下进行批量编译

ROS NODE

  • 一个节点是ROS程序包中的一个可执行文件
  • ROS节点可以使用ROS客户库与其他节点通信
  • 节点可以发布或者接受一个话题
  • 节点可以提供或使用某项服务

常用命令：

  • rosnode list：查看当前注册到ros master的所有节点
  • rosnode info：查看某个节点的具体信息

ROS TOPIC

节点之间是通过一个ROS topic来相互通信的。通过publisher声明发topic名称，通过subscriber声明所需要监听的topic名称

常用命令：

  • Topic list：查看当前注册到ros master的所有topic列表
  • rostopic echo：把当前topic输出到控制台，方便调试和查看

ROS PARAM

ROsparam命令允许我们在ROS参数服务器上存储和复制数据。参数服务器可以存储整数、浮点数、布尔值、字典和列表值。

常用命令：

  • rosparam set：设置参数
  • rosparam get：获取参数
  • rosparam list：查看当前服务器上的参数
